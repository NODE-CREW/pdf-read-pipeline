# Code Quality Review

## 목적
- 증상 완화가 아닌 근본 원인 해결 여부를 확인한다.
- 회귀 가능성을 사전에 줄인다.

## 입력
- 변경 파일 목록 또는 diff
- 관련 요구사항/이슈 맥락
- 기존 테스트 현황

## 절차
1. 변경 의도와 실제 구현의 일치 여부를 확인한다.
2. 동일 기능의 기존 함수/모듈 존재 여부를 먼저 찾는다.
3. 중복 구현이 있으면 재사용/통합 가능성을 우선 검토한다.
4. 실패 경로(예외, 빈 입력, 경계값)를 중심으로 버그 가능성을 점검한다.
5. 테스트가 변경 리스크를 충분히 방어하는지 평가한다.

## 우선순위 규칙
- `P1`: 데이터 손실, 보안, 서비스 중단 가능성
- `P2`: 기능 오동작/회귀 가능성이 높음
- `P3`: 유지보수성 저하, 구조적 부채 증가

## 출력 형식
- `Findings`: 심각도 순 정렬
- 각 항목 필수 포함: 파일, 라인, 문제 요약, 근거, 권장 수정
- `테스트 갭`: 추가해야 할 테스트 항목
- `가정/오픈 이슈`: 확정 불가 사항

## 작성 원칙
- 칭찬/개요보다 결함 식별을 우선한다.
- 일반론보다 실제 변경 코드 근거를 제시한다.
- 수정 제안은 최소 변경으로 효과가 큰 순서로 제시한다.
